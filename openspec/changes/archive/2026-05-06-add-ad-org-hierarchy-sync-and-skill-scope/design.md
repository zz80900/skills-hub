## Context

当前系统的 AD 集成只在登录时通过 Kerberos + LDAP 校验身份，并把 `display_name` 与 `external_principal` 同步到本地用户表。LDAP 查询阶段已经能拿到 `distinguishedName` 与完整属性集合，但本地 `users` 表没有组织架构字段，因此系统既无法展示用户所在部门，也无法基于 AD 组织做 Skill 可见范围控制。

另一方面，系统已经存在基于 `group_id` 的 Skill 组范围可见性规则，前后端也已经围绕“管理员可绑定任意组、普通用户只能绑定自己所在组、首页和详情按组过滤”建立起完整链路。本次变更必须在不破坏现有组能力的前提下，引入“组织范围”这一第二种可见性维度，并保证未来 AD 组织调整后，用户下次登录即可自动生效。

当前仓库仍采用 `Base.metadata.create_all()` 加 `ensure_schema_compatibility()` 的方式兼容升级，没有 Alembic，因此设计需要兼顾 SQLite / PostgreSQL 两种运行环境，并尽量避免复杂迁移。

## Goals / Non-Goals

**Goals:**
- 在 AD 用户每次成功登录时，从 `distinguishedName` 解析出最多 4 级组织架构并同步到本地。
- 去掉全体用户共享的顶级根节点，只保留可用于业务判断的 OU 路径，例如 `OU=新国都集团,DC=xgd,DC=com` 之上的共同根不进入组织层级。
- 支持个人中心展示当前用户的组织架构，兼容少于 4 级的场景。
- 为 Skill 增加与现有组范围兼容的组织范围能力，并允许普通用户仅绑定自己当前所在路径中的末级组织。
- 当 Skill 绑定到某级组织时，允许该组织下所有更深层级的子组织用户查看该 Skill。

**Non-Goals:**
- 不把 AD 组织架构扩展为独立组织树表，也不提供后台手工维护组织树能力。
- 不支持一个 Skill 同时绑定多个组或多个组织节点。
- 不改造 Skill 的 workspace 所有权；组织可见不会授予编辑、升级、删除权限。
- 不处理本地用户的人工组织录入；组织同步仅针对 AD 来源用户。

## Decisions

### 1. 在 `users` 表中缓存扁平化组织层级，而不是新增独立组织节点表

本次选择在 `users` 表直接新增以下字段：

- `ad_distinguished_name`
- `org_level_1`
- `org_level_2`
- `org_level_3`
- `org_level_4`
- `org_path`
- `org_depth`

其中：

- `ad_distinguished_name` 保存最近一次登录同步到的原始 DN，便于排查问题。
- `org_level_1` 到 `org_level_4` 保存从高到低的组织名称。
- `org_path` 保存标准化路径字符串，例如 `支付硬件事业群 / 技术中心 / 公共技术中心 / 系统方案部`。
- `org_depth` 保存当前用户实际有效层级数。

不引入独立组织树表的原因：

- 当前需求只要求“按登录实时同步”和“按路径继承判断可见性”，并不要求后台维护、跨用户汇总报表或组织树审计。
- 扁平字段可以直接支撑个人中心展示、末级组织选择和可见性匹配，数据模型更简单，符合 KISS / YAGNI。
- 在没有迁移框架的前提下，新增扁平字段的兼容升级成本更低。

备选方案：

- 单独创建 `organizations` 表并维护 parent-child 关系：更规范，但需要额外处理去重、组织变更、孤儿节点清理和同步幂等，超出当前需求。

### 2. 组织架构解析基于 AD 用户 DN 的 OU 段，并裁掉共享根节点

组织架构解析规则定义为：

1. 从 `distinguishedName` 中提取所有 `OU=` 段，忽略 `CN=` 与 `DC=`。
2. 按 DN 原始顺序从叶子到根读取 OU，例如：
   - `OU=系统方案部`
   - `OU=公共技术中心`
   - `OU=技术中心`
   - `OU=支付硬件事业群`
   - `OU=新国都集团`
3. 将 OU 列表反转为“高到低”的组织路径。
4. 裁掉共享根节点，默认规则为：如果 OU 数量大于 1，则丢弃最顶层 OU，仅保留其下最多 4 级。
5. 若裁剪后层级超过 4，只保留最靠近用户的 4 级中的高到低顺序；若不足 4，则剩余字段置空。

以 `CN=谢金城,OU=系统方案部,OU=公共技术中心,OU=技术中心,OU=支付硬件事业群,OU=新国都集团,DC=xgd,DC=com` 为例，最终结果是：

- `org_level_1 = 支付硬件事业群`
- `org_level_2 = 技术中心`
- `org_level_3 = 公共技术中心`
- `org_level_4 = 系统方案部`

这里把“裁掉共享根节点”固化为“丢弃最顶层 OU”的原因是：用户给出的业务规则已经明确说明 `新国都集团` 不进入系统的可用组织架构。相比引入可配置根 OU 白名单，这一规则更直接，且足以覆盖当前域结构。

备选方案：

- 从配置中读取根 OU 名称：更灵活，但会增加部署复杂度，当前没有明确多租户或多根节点需求。
- 保留完整 OU 路径再由前端裁剪：会让后端权限判断和前端展示使用不同语义，容易出现不一致。

### 3. AD 组织同步放在每次成功登录后的用户同步阶段执行

当前 AD 用户登录已经走 `authenticate_active_directory_user()` -> `provision_ad_user()` / `sync_ad_user_profile()` 链路。本次在同一同步阶段增加组织字段写入，规则为：

- 首次 AD 登录：创建本地用户时一并写入组织字段。
- 已存在 AD 用户再次登录：每次都根据最新 DN 重算组织字段并覆盖旧值。
- 本地用户登录：不触发任何组织同步逻辑。

这样设计的原因：

- 登录本身已经是域信息可信的最新时刻，天然适合同步组织缓存。
- 不需要额外定时任务或后台批量同步机制。
- 能满足“组织架构调整后下次登录立即生效”的要求。

备选方案：

- 单独做后台同步任务：组织变更传播更快，但会引入计划任务、失败补偿和域访问负载问题。

### 4. Skill 增加统一的“范围类型 + 范围目标”模型，兼容现有组范围

Skill 范围模型调整为：

- `scope_type`: `PUBLIC | GROUP | ORGANIZATION`
- `group_id`: 当 `scope_type = GROUP` 时使用
- `scope_org_level`: 1 到 4，表示绑定的是哪一级组织
- `scope_org_name`: 绑定组织的名称
- `scope_org_path`: 从 1 级到目标层级的标准化路径

规则如下：

- `PUBLIC`：当前公开行为，不限制登录用户身份。
- `GROUP`：沿用现有 `group_id` 规则。
- `ORGANIZATION`：按 `scope_org_path` 与用户组织路径匹配。

不直接只保存“末级组织名称”的原因是同名部门可能出现在不同分支下，必须连同层级路径一起保存，才能稳定判断“同一路径或子路径”可见。

备选方案：

- 只新增 `org_path`，不保留 `scope_org_level`：也能工作，但前后端展示和调试时不够直观。
- 使用多态关联表承载范围目标：模型更抽象，但对当前单目标场景过度设计。

### 5. 组织范围的可见性采用前缀匹配，表达“父部门对子部门可见”

当 Skill 绑定组织范围时，访问判定规则为：

- 管理员始终可见。
- 用户必须是 AD 用户，且有非空组织路径。
- 若用户的 `(org_level_1 ... org_level_n)` 前 n 级与 Skill 的 `scope_org_path` 完全一致，则可见，其中 n 为 Skill 绑定层级。

例如：

- Skill 绑定到 3 级 `支付硬件事业群 / 技术中心 / 公共技术中心`
- 用户为 3 级该部门本身，可见。
- 用户为 4 级 `支付硬件事业群 / 技术中心 / 公共技术中心 / 系统方案部`，也可见。

这样做能自然表达“3 级部门上传的 Skill，4 级子部门同事也能看到”，同时不会把兄弟部门误判为可见。

### 6. 普通用户只能绑定自己当前末级组织，管理员可以选择任意存在的组织路径

前端 Skill 表单会新增范围类型选择：

- 公开可见
- 归属组可见
- 归属组织可见

当选择组织可见时：

- 普通用户只能看到自己当前组织路径的末级节点作为唯一可选项。
- 管理员可以从系统现有 AD 用户组织路径聚合出的组织节点列表中选择任意节点。

这里选择“普通用户只能绑末级组织”是因为需求已经明确限制，且能避免普通用户把 Skill 上提到更大部门范围。管理员保留全量选择能力，用于运营场景下代发或调整范围。

备选方案：

- 普通用户可在自己路径上任选祖先节点：灵活但违反当前业务约束。

## Risks / Trade-offs

- [仅缓存用户组织字段，不维护独立组织树] → 适合当前需求，但后续若要做组织管理后台，需要再抽象组织实体。
- [裁掉最顶层 OU 的规则依赖当前域结构] → 先将规则写入 spec 与实现；若后续域结构变化，再平滑扩展为配置项。
- [管理员选择任意组织节点需要从现有用户路径聚合] → 若某个组织暂时没有任何活跃 AD 用户，本期无法被管理员选中；当前需求未要求预建空组织节点。
- [组织范围与组范围并存后，Skill 表单更复杂] → 通过单选范围类型和条件显示目标输入，保持交互简洁。
- [本地用户没有组织路径] → 组织范围 Skill 对这类用户默认不可见，这是符合预期的。

## Migration Plan

- 为 `users` 表补充组织字段，为 `skills` 表补充 `scope_type`、组织范围字段，并保留 `group_id` 以兼容现有组逻辑。
- 对历史 Skill 执行一次兼容迁移：
  - `group_id IS NULL` 的记录回填为 `scope_type = PUBLIC`
  - `group_id IS NOT NULL` 的记录回填为 `scope_type = GROUP`
- 前端在旧数据缺失组织字段时按空值展示，不阻塞页面加载。
- 回滚时保留新增列不会影响旧代码读取；只需确保旧版本忽略这些列即可。
